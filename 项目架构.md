# QYAI AC79 A100 SDK 项目阅读手册

## 1. 项目定位

这个仓库本质上是杰理 AC79/WL82 SDK 的一个定制工程，当前保留的业务核心是轻语 AI 的 `wifi_story_machine` 方案。  
它不是一个从零搭建的纯应用仓库，而是“公版 SDK + 板级配置 + 业务应用层 + 预编译库 + 打包烧录工具”的组合体。

对后续开发最重要的判断是：

- `apps/wifi_story_machine` 是当前主要业务工程。
- `apps/common` 是大量通用能力的来源，包含音频、网络、BLE、UI、升级、设备管理等公共模块。
- `cpu/wl82/liba` 和 `lib/` 中有大量预编译库，很多底层或平台能力不在源码里。
- `cpu/wl82/tools` 是编译后处理、资源打包、烧录、OTA 文件生成的核心目录。

## 2. 当前仓库的实际状态

仓库根目录仍然保留了上游 SDK 的 `Makefile` 和 `default.workspace`，但当前代码树已经裁剪，不再包含全部示例工程。

当前 `apps/` 下实际只有：

- `apps/common`
- `apps/demo`
- `apps/wifi_story_machine`

因此要注意：

- 根目录 `Makefile` 里的 `all` 目标引用了很多当前仓库不存在的工程，不能直接当成“当前仓库完整可用”的入口。
- 目前应该优先使用 `apps/wifi_story_machine/board/wl82/Makefile` 或根目录的单独目标 `make ac791n_wifi_story_machine`。
- `default.workspace` 里的很多 `.cbp` 工程路径也是上游遗留，并不都对应现有目录。

结论：这个仓库现在是“围绕 `wifi_story_machine` 的定制 SDK 子集”，不是完整的 AC79 全量发布包。

## 4. 架构分层理解

可以把这个项目理解成 5 层：

1. 芯片与 SDK 平台层  
   `cpu/wl82`、`include_lib`、`cpu/wl82/liba`

2. SDK 通用能力层  
   `apps/common`

3. 板级适配层  
   `apps/wifi_story_machine/board/wl82`

4. 业务应用层  
   `apps/wifi_story_machine/app_main.c` 和 `apps/wifi_story_machine/chatbot`

5. 资源与交付层  
   `cpu/wl82/tools`、`ui_res`、`audlogo`、OTA 打包脚本

后续开发时，优先判断你的修改属于哪一层，不要把所有需求都直接塞进 `app_main.c`。

## 5. 主工程启动流程

主入口在 `apps/wifi_story_machine/app_main.c`。

从代码结构看，启动流程大致是：

1. 输出启动日志，进入 `wifi_story_machine` 应用。
2. 做 Flash 保护校验，防止固件与外部 Flash 配置不匹配。
3. 判断是否进入产测模式。
4. 做电池电量检查、软关机恢复判断。
5. 初始化 UI/表情、闹钟显示、时间显示等外围表现层。
6. 读取网络模式与网络通道配置。
7. 按配置决定启动 WiFi 还是 LTE。
8. 初始化蓝牙、LED、G-sensor 等外设能力。
9. 最后调用 `audio_app_mode_switch(NULL)` 进入默认音频/业务模式。

这个入口同时还定义了：

- `irq_info_table`
- `task_info_table`
- 默认事件处理

也就是说它不仅是“应用入口”，还是这个方案的系统集成点。

## 6. 任务模型

`app_main.c` 中的 `task_info_table` 很关键，它能直接说明这套固件的运行模型。

当前能看到的主要任务类别有：

- 系统任务：`app_core`、`sys_event`、`sys_timer`、`systimer`
- 音频任务：`audio_server`、`audio_mix`、各种 encoder
- 网络任务：`tcpip_thread`
- WiFi 任务：`tasklet`、`RtmpMlmeTask`、`RtmpCmdQTask`、`wl_rx_irq_thread`
- 蓝牙任务：`btctrler`、`btstack`、`btencry`
- UI 任务：`ui`、`lcd_task_0`、`lcd_task_1`、`te_task`
- 视频任务：`video_server`、`video_dec_server` 等
- 升级任务：`update`、`dw_update`
- AIUI 任务：`aiui_net_task`
- 4G 任务：启用 LTE 宏后会出现 `lte_rx_task`

这说明它不是单线程的“普通单片机逻辑”，而是一个带 RTOS 任务调度的复杂应用系统。  
改功能时要考虑任务上下文、同步方式、是否允许阻塞。

## 7. 关键配置入口

### 7.1 板型与工程档位

文件：`apps/wifi_story_machine/board/wl82/board_config.h`

这里决定两件事：

- 选哪个芯片/封装板型，例如当前默认是 `CONFIG_BOARD_7916AB`
- 选哪个工程配置档位，例如当前默认是 `CONFIG_BOARD_DEVELOP`

如果后续换板、换封装、换 UI 板、换 4G 板，首先看这里。

### 7.2 开发板功能宏

文件：`apps/wifi_story_machine/board/wl82/board_7916A_develop_cfg.h`

这个文件是当前最重要的板级配置文件之一。  
当前默认明显偏向轻语开发板：

- `SXY_QYAI_DEVELEP_BOARD = 1`
- `SXY_QYAI_4G_DEMO_BOARD = 0`
- `SXY_QYAI_UI_BOARD = 0`

这个文件里会决定：

- 是否启用 4G
- 是否启用 UI
- 是否启用蓝牙/SD/USB/AUX 模式
- 本地唤醒参数
- 调试串口
- 电源控制脚
- LED、按键、DAC mute、SD 卡、TM1629、WS2812 等引脚

大部分“跟硬件有关但又不属于驱动源码”的改动，都应该先看这里。

### 7.3 应用级总配置

文件：`apps/wifi_story_machine/include/app_config.h`

这个文件更偏全局功能开关，负责：

- 网络能力开关
- 音频能力开关
- UI 资源策略
- 升级/分区策略
- 本地唤醒算法选择
- BLE 名称、BT 名称
- 文件系统/存储相关配置

值得特别注意的点：

- 默认启用了 `CONFIG_ASR_ALGORITHM  IFLYTEK_ALGORITHM`
- 也保留了 `KWS_QYAI_ALGORITHM`、`KWS_YUNZS_ALGORITHM` 等可选项
- `BLE_DEV_NAME` 默认前缀是 `QYAI_AC_`
- `BT_DEV_NAME` 默认前缀是 `QYAI_`

如果要切换唤醒算法或设备对外标识，这里是第一入口。

### 7.4 轻语服务器参数

文件：`apps/wifi_story_machine/chatbot/qyai_config.c`

这里负责：

- 服务端环境选择：正式/测试/预发
- AI 音频编码格式
- AI 音频解码格式
- 语言选择
- 网络异常回调

后续如果要切换轻语服务地址、调整音频编解码策略，先看这里。

## 8. 业务模块拆解

### 8.1 `chatbot/audio/`

这是最核心的业务目录之一，负责：

- `ai_speaker.c`
  AI 对话模式主逻辑，本地唤醒、联网对话、WiFi/LTE 协同、恢复出厂、配网/睡眠等行为都和它密切相关。
- `mode.c`
  模式切换。
- `music_play.c`
  提示音和本地音频播放基础能力。
- `net_music.c`
  网络音乐播放。
- `sd_music.c`、`usbdisk_music.c`、`aux_music.c`
  本地介质/外部输入模式。
- `mp3_buf_play.c`
  缓冲播放。
- `volume.c`
  音量控制。
- `recorder.c`
  录音相关。

如果需求和“对话、播报、音乐、提示音、模式切换”有关，优先从这里切入。

### 8.2 `chatbot/ai_uart_ctrol.*`

这个模块是轻语 AI 串口协议的关键实现，对应 `doc_qyai/AI大模型模组通用-串口协议v1.5.txt`。

它主要承担：

- AI 模组和主控之间的命令映射
- 表情状态码、对话状态码、音乐控制码
- WiFi/蓝牙/电量/设备控制命令

如果后续要对接外部 AI 模组、屏幕模组、协处理器，这是必须看的入口。

### 8.3 `chatbot/AT_UART_CMD.*`

这是另一套串口协议封装，更偏 AT 风格输出。  
从代码看，它把内部事件转换成如 `AI_START`、`TTS=...`、`MUSIC_PLAY` 这类文本指令。

适用场景：

- 外部串口联调
- 模块化联机
- 产测或第三方 MCU 联动

### 8.4 `chatbot/user_private_cmd.c`

这个文件很像“产品级自定义命令和对话行为中枢”。  
里面处理了很多业务规则：

- 用户结束对话/休眠类语句判断
- 睡眠场景 TTS 和白噪音逻辑
- LTE 配对码提取
- 对话文本清洗
- UI 页面恢复
- 网络对话文本回调

如果以后要加“语义后处理”“产品私有口令”“特定场景播报”，优先改这里。

### 8.5 `chatbot/time/`

- `alarm.c`
  闹钟逻辑
- `tm1629.c`
  数码显示驱动/显示控制
- `time_uart_calibration.c`
  串口时钟校准

### 8.6 `chatbot/lte/`

- `usb_net_at_cmd.c`
  4G 模组 USB AT 管理、厂商识别、SIM/联网检查
- `lte_wifi_config.c`
  4G/WiFi 配置协同

从代码可见，已适配多个 4G 模组厂商，至少包含：

- ML307R/C
- AIR780E
- Yuge YM310
- ML120H

### 8.7 `chatbot/http/` 与 `chatbot/mqtt/`

- `http_more_tts_play.c`
  网络 TTS/播放扩展
- `mqtt_client.c`

说明这个方案除了 WebSocket/HTTP 对话，也保留了 MQTT 设备信息链路。

### 8.8 `chatbot/printer/`

目录里有：

### 8.9 `iflytek/`

### 8.10 `ui/`

## 9. 网络与配网

### 9.1 网络通道

`app_main.c` 中明确存在网络通道选择逻辑：

- WiFi
- LTE
- 自动模式/手动模式

默认看起来更偏 WiFi 优先，LTE 作为可选通道。

### 9.2 配网方式

`doc_qyai/配网/配网-交互过程(外部).md` 说明当前方案使用 BLE 配网：

- 设备作为 BLE Peripheral 广播
- 手机连接后通过 GATT 写入 WiFi SSID/密码
- 数据使用自定义协议，头部 `JL`
- 设备回传配网状态码

这意味着如果后续要改配网体验，要同时关注：

- `apps/common/ble/`
- `apps/wifi_story_machine` 业务层
- 文档里的协议兼容性

### 9.3 4G 扩展

`doc_qyai/4G-LTE模组/4G-LTE模组移植说明(AC792N).md` 给出的要点非常明确：

- 应用层需要接入 LTE 初始化逻辑
- 依赖 `lte_module.a`
- 需要开启 LTE 相关宏
- 产品名称需要带 `4G` 后缀

这说明 4G 不是完全独立的另一个工程，而是同一业务方案的变体。

## 10. 编译、打包、烧录链路

## 10.1 Windows 工具链

`apps/wifi_story_machine/board/wl82/Makefile` 里写死了 Windows 工具链位置：

- `C:/JL/pi32/bin`

主要工具：

- `clang.exe`
- `pi32v2-lto-wrapper.exe`
- `llvm-ar.exe`

如果这套路径不存在，编译不会成功。

## 10.2 推荐构建入口

### 方式一：在根目录构建单工程

```bash
make ac791n_wifi_story_machine
```

### 方式二：直接进主工程板级目录

```bash
cd apps/wifi_story_machine/board/wl82
make
make clean
```

不建议直接执行根目录 `make all`，因为当前仓库缺少上游完整工程集。

## 10.3 编译产物

板级 `Makefile` 和 `cpu/wl82/tools` 可以看出主要产物路径在：

- `cpu/wl82/tools/sdk.elf`
- `cpu/wl82/tools/app.bin`
- `cpu/wl82/tools/jl_isd.fw`
- `cpu/wl82/tools/jl_isd.ufw`
- `cpu/wl82/tools/db_update_files_data.bin`
- `cpu/wl82/tools/packres/AUPACKRES`
- `cpu/wl82/tools/packres/UIPACKRES`

## 10.4 编译后处理

构建后的核心动作不是只生成 ELF，还会执行后处理脚本：

- 提取 `text/data/ram` 段生成 `app.bin`
- 打包 UI 资源为 `UIPACKRES`
- 打包提示音资源为 `AUPACKRES`
- 调用 `isd_download.exe`
- 生成 `.fw` 和 `.ufw`

也就是说：  
这个工程的“编译完成”实际上包含了“资源打包 + 下载/升级文件生成”。

## 10.5 烧录与升级

`cpu/wl82/tools` 中有完整工具链：

- `isd_download.exe`
- `ufw_maker.exe`
- `SM01-DFU.exe`
- `生成升级固件.bat`
- `生成升级固件-OTA.bat`
- `生成升级固件-OTA-RES预留区资源.bat`

如果只是开发调试，通常直接用构建后的下载流程即可。  
如果要交付 OTA 包或单独资源升级包，重点看这些批处理脚本。

## 11. 资源组织与 OTA 风险

这是这个 SDK 最容易踩坑的部分。

从 `download.bat`、`download-res.bat`、`isd_config_rule.c`、`note.txt`、`OTA升级流程.txt` 可以确定：

- 音频提示音和 UI 资源会被打包进预留区资源包。
- 资源包地址和长度与 OTA 升级强相关。
- 一旦量产烧录，`AUPACKRES_ADR/LEN`、`UIPACKRES_ADR/LEN` 不能随意变更。
- 跨版本升级时，`isd_config_rule.c` 里的分区配置必须保持兼容。

后续如果改：

- 提示音
- UI 资源
- Flash 布局
- OTA 策略

必须同步检查：

- `cpu/wl82/tools/isd_config_rule.c`
- `cpu/wl82/tools/download*.bat`
- `cpu/wl82/tools/note.txt`
- `cpu/wl82/tools/OTA升级流程.txt`

这部分配置错了，最常见结果不是“编译不过”，而是“能烧录但升级失败、资源错位、升级后起不来”。

## 12. 预编译库边界

`cpu/wl82/liba` 里有大量预编译库，说明很多能力不是源码开放的。

和当前项目关系密切的包括：

- `sxy_qyai.a`
- `sxy_qyai_nokw_sfc.a`
- `sxy_qyai_sfc.a`
- `lte_module.a`
- `ai_server.a`
- `audio_server.a`
- `net_server.a`
- `btstack.a`
- `btctrler.a`
- `ui.a`
- `lwip_2_1_3*.a`
- `libifly_aiui_lib.a`
- `lib_mqtt.a`

开发上的实际含义：

- 业务层可以改。
- 调用方式可以改。
- 参数、流程、外围逻辑可以改。
- 但如果问题落在库内部实现，就只能绕过、替代、切换配置，或者更换库版本。

后续排障时要先判断问题属于“源码层”还是“库黑盒层”。

## 13. 后续开发最常用的改动入口

### 13.1 改唤醒词、对话行为、私有语义

优先看：

## 15. 我对这个项目的结论

这不是“普通聊天机器人应用”，而是一套完整的嵌入式 AI 终端方案：

- 底座是杰理 AC79/WL82 SDK
- 业务主线是 `wifi_story_machine`
- 本地唤醒、云端对话、音乐播放、BLE 配网、4G 扩展、UI 表情、提示音资源、OTA 升级都已经串在一起
- 很多平台能力来自预编译库，源码层主要负责集成和产品逻辑

因此后续开发的正确方式应该是：

- 先判定需求属于哪一层
- 优先在 `wifi_story_machine` 业务层收敛改动
- 需要共性能力时再进入 `apps/common`
- 改资源和 OTA 时必须同步检查 `cpu/wl82/tools`
