# ADR 0001：优先在业务层改动，并明确预编译库边界

- 状态：Proposed
- 日期：2026-07-03

## 背景

这个仓库不是一个小而完整的独立固件应用，而是更大范围的杰理 AC79/WL82 SDK 的一个定制化子集，主要包含：

- `apps/wifi_story_machine` 中的产品业务代码
- `apps/common` 中的可复用通用源码
- `apps/wifi_story_machine/board/wl82` 中的板级配置
- `cpu/wl82/liba` 和 `lib` 中的大量预编译库
- `cpu/wl82/tools` 中耦合紧密的后处理、资源打包和 OTA 工具链

正因为结构如此分层，开发时很容易在错误的层次上改动：

- 明明只是当前产品的需求，却去改共享代码
- 明明真实控制点在板级宏或功能开关，却在源码行为里反复打补丁
- 明明问题可能落在预编译库内部，却误判成源码层问题
- 改了资源或 OTA 相关文件，却没有同时检查打包和交付后果

## 决策

以后在实现或调试产品行为时，默认遵循下面的顺序：

1. 优先在 `apps/wifi_story_machine` 业务层落改动。
2. 只有当行为确实属于多产品共享能力时，才进入 `apps/common`。
3. 只要问题涉及硬件、能力开关或变体配置，就要尽早检查板级配置和应用功能宏。
4. 预编译库默认视为黑盒边界，除非已经有明确证据证明所需改动落在源码层。
5. 只要涉及 `cpu/wl82/tools`，就把它视为“影响交付”的改动，并与资源包和 OTA 后果一起审视。

## 影响

### 正面影响

- 产品行为改动默认保持在本地，影响范围更可控。
- 共享 SDK 代码不容易被产品私有逻辑污染。
- 调试时更不容易在预编译库黑盒里空转。
- 资源包和 OTA 风险会更早暴露出来。

### 负面影响

- 有些修复在前期会多花一点时间，因为必须先判断责任层级。
- 真正属于共享能力的改动，可能会先在业务层试做，再回提到公共层。
- 遇到库内问题时，可能需要设计绕行方案，而不是直接修库。

## 备注

这条 ADR 目前作为后续维护的默认基线使用。如果未来仓库中有更多原本由预编译库承担的能力转为源码可控，应当回头更新这条决策。
